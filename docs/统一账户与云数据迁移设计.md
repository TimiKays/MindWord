# MindWord 统一账户与云数据迁移设计

## 1. 当前结论

MindWord 将从 Supabase Auth + `user_data` 迁移到 TimiAuth + TimiCloud。迁移以直接导入为主，首次登录迁移只处理异常账号。

2026-07-22 全量迁移记录：

- Supabase 最新只读快照：104 个账号、27 份 `user_data`，与写后再次核对的实时数量一致。
- 全量迁移：102 个账号新建，2 个已迁内部账号只刷新云数据且不改密码；共建立 104 条身份映射、写入 27 份 `workspace_v1` 和 189 个分片。
- 密码哈希、邮箱验证状态和云数据均已写后核验；14 个旧系统未验证账号仍保持未验证，必须正常走验证码流程。
- `feedback`：0 行，不迁移；Supabase 源数据仍完整保留。

已完成内部试用开关、TimiAuth 登录状态接入和 TimiCloud `workspace_v1` 适配。默认模式仍完整使用 Supabase；只有当前标签页显式开启内部模式才加载统一账户组件。同步继续沿用原有的逐项预览和人工选择，不会用云端静默覆盖本地文档；AI 密钥上传前会递归剥离，选择云端配置时会把本机密钥补回。

生产已执行统一数据库迁移 0010 / 0011，并在账号所有者授权下完成全量导入。写前备份、隔离首次/重复执行演练、写后备份和逐项验收均通过；后端完整测试 188 项通过。MindWord 内部统一模式已可使用，但默认用户仍走 Supabase，数据迁移完成不代表默认登录和同步链路已经切换。

## 2. 用户体验

迁移后用户继续使用原邮箱和原密码。旧密码明文不会被读取或导出，统一系统直接复用兼容的 bcrypt 哈希。

- 新系统已有账号且密码正确：直接登录。
- 新系统已有账号但密码错误：直接失败，绝不尝试旧密码接管新账号。
- 新系统不存在、旧哈希兼容：账号随生产迁移预先创建，用户直接登录。
- 密码哈希缺失或不兼容：首次登录验证旧系统，成功后创建账号。
- 同邮箱在新旧系统同时存在：不覆盖新密码，进入单独身份核验和旧数据绑定。
- 账户或云服务故障：本地编辑、导入和导出始终可用。

MindWord 只要求重新建立本产品的云同步状态，不让 MarkPage、myZone 等其他产品一起退出。

## 3. 数据合同

TimiCloud 使用：

- `product=mindword`
- `data_key=workspace_v1`

`workspace_v1` 保存现有同步所需的完整工作区：

- `docs`
- `aiConfig`（剥离 API Key、token、密码和私钥）
- `promptTemplates`
- `myPromptTemplates`
- 文档、配置和模板更新时间
- 配置与模板哈希

Base64 图片首期继续跟随文档快照迁移。单条数据必须低于服务端 10MB 限制，并预留 64KB 请求空间；超限账号不静默丢弃，进入异常清单。D1 单字段小于该上限，统一云存储已在 0011 中增加透明分片：前端仍读写一份 `workspace_v1`，底层自动拆分、校验和还原。

## 4. 迁移规则

每个旧账号都必须通过 Supabase `user_id` 建立到统一 `user_id` 的映射。不得只凭前端传入的邮箱或用户 ID 写入云数据。

直接迁移时同时准备：

1. 统一账号和原 bcrypt 哈希。
2. 邮箱验证状态和创建时间。
3. MindWord 产品开通记录与免费会员记录。
4. `legacy_identity_mappings` 身份映射。
5. 脱敏后的 `workspace_v1` 和存储配额。

迁移脚本必须可重复运行。请求超时或重复执行不能生成第二个账号、第二条映射或第二份工作区。

## 5. 冲突与回退

- 新系统同邮箱账号：不覆盖密码；只有显式一对一绑定清单通过核验后，才迁移旧云数据。
- 旧数据在导出后发生变化：重新生成该账号的迁移计划，不用旧快照覆盖。
- 迁移失败：保持 Supabase 可用，不删除旧数据。
- 内部灰度失败：关闭 MindWord 新入口，恢复旧 Supabase 读取。
- Supabase 至少保留 30 天，稳定期结束后再决定退役。

## 7. 当前下一步

账号和云数据的全量迁移与旧账号登录、云同步验收已经完成。默认 `/app` 发布后进入统一账户和统一云端；`?account_mode=legacy` 只用于短期故障回退。旧 Supabase 数据不删除，也不会被新入口覆盖。
